## Key methodology / process notes for the next LLM

### Working with the manuscript Gdoc

- **Direct edits via Docs API**: works for insertions/deletions but cannot create tracked-change suggestions. Use this only for hard-confirmed changes (e.g., the C14 references — JC explicitly said "you can just post directly").
- **Tracked-change suggestions**: only achievable via the UI in Suggesting mode. The Claude in Chrome extension can drive this, but with caveats:
  - The doc body is canvas-rendered. Text selection via DOM/JS clicks does NOT work; use `Ctrl+F` to navigate to anchor text, `Esc`, then arrow keys (Right/Left) to position cursor for insertion.
  - `Ctrl+H` Find & Replace works reliably for replacements/deletions. In Suggesting mode, F&R creates tracked-change suggestions automatically.
  - For multi-paragraph inserts, the `type` action via the computer tool times out on very long strings (>~700 chars). Split into chunks, with `Return` keypresses between paragraphs.
  - Long-pattern F&R via the `__doReplace` JS helper (installed in the page; defined in `post_v3_comments_to_manuscript.py` lineage) is faster and more reliable than typing into the F&R dialog character-by-character.
  - Footnotes via `Ctrl+Alt+F` do NOT work — that combo registers as a literal `F`. Insert menu does not surface "Footnote" directly. Add footnotes manually.
- **Drive API anchored comments do NOT bind on Docs**. We tried both `{"r":"head","a":[{"txt":...}]}` JSON anchors and plain `kix.xxx` strings (created via NamedRange). Both produce "Original content deleted" orphan comments. Don't waste time on this path; use Suggesting mode or unanchored comments.

### Working with the Chrome extension

- Tab freezes occasionally on long doc operations. Wait or have JC click into the tab to wake it.
- DPR scaling: screenshot coords are about 0.77× the actual viewport coords. For coordinate clicks, use the `getBoundingClientRect().x/y` values returned from JS, not the screenshot pixel space.
- Synthetic clicks via JS `dispatchEvent` need the FULL chain: `pointerdown, mousedown, pointerup, mouseup, click`. A bare `.click()` is filtered by Docs.
- Setting input values via React-bypassing native setter: use `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set.call(input, value)` then dispatch an `input` event. Plain `.value = ...` does not trigger React state.
